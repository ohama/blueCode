# Milestone v1.1: blueCode Refinement

**Status:** ✅ SHIPPED 2026-04-24
**Phases:** 6-7
**Total Plans:** 5 (3 in Phase 6 incl. 06-03 gap closure; 2 in Phase 7)

## Overview

v1.1 cleaned up three pieces of technical debt surfaced during v1.0 UAT: a hardcoded absolute-path in `Router.modelToName` that broke portability, a bootstrap-time probe that fired even when the user targeted a different model port, and a `Step.Thought` placeholder that made `--verbose` output meaningless. A mid-milestone live regression revealed a subtle HF-fallback trap in `mlx_lm.server` that REF-01's initial implementation triggered — resolved by a targeted gap closure plan (06-03). Post-fix, live verification confirmed all three requirements working end-to-end: blueCode sends the local model path in POST body, bootstrap touches no network, and `--verbose` displays actual LLM reasoning in `Step.Thought`.

## Milestone Summary

**Decimal Phases**: None inserted in v1.1.

**Gap Closure Plans**:
- Phase 6 added 06-03 after live re-verification caught the `tryParseModelId` HF-id regression (blueCode was sending `data[0].id` which mlx_lm.server resolved against HuggingFace Hub, fetching Base Coder tokenizer and overwriting the loaded Instruct template — chat template effectively lost, responses became FIM continuations)

### Key Decisions

**Phase 6:**
- **Option B over A/C** (06-RESEARCH): Delete `Router.modelToName` entirely; adapter-layer `QwenHttpClient` owns wire value via `ModelInfo` record injected from lazy probe. Core purity preserved.
- Two explicit `Lazy<Task<ModelInfo>>` (probe8000, probe8001) rather than `Map<Model, Lazy>` — named bindings clearer for two-model enum
- `CancellationToken.None` inside `Lazy` factory — shared task across multiple callers must not be cancelled by one caller's Ctrl+C
- `AppComponents.MaxModelLen` stays `int = 8192` floor (per-port actual lives inside QwenHttpClient lazy probe; not surfaced to AppComponents) — v1.2 candidate for per-port accuracy
- `getMaxModelLenAsync` fully removed in 06-02 (not soft-aliased) — no caller after `bootstrapAsync` deletion; `probeModelInfoAsync` supersedes it
- Test port 64321 (deterministically closed) replacing hardcoded 8000 for closed-port fallback test
- **06-03 gap closure**: `tryParseModelId` prefers `id.StartsWith("/")` (local path) with `List.tryHead` fallback for single-id servers. Heuristic is macOS/Linux-only (v1 Mac-only constraint permits this).

**Phase 7:**
- **Option C over A/B/D** (07-RESEARCH): New Core record `LlmResponse = { Thought: Thought; Output: LlmOutput }` in Domain.fs — keeps `LlmOutput` pattern matches unchanged everywhere, gives `callLlmWithRetry` readable named return type, respects Core purity (no `JsonElement`, no `LlmStep` in Core)
- Big-bang single-task compile cascade — F# type system forces all 4 production files into one atomic commit; intermediate state fails to compile so incremental migration unnecessary
- `makeMockResponse` helper duplicated in AgentLoopTests.fs + ReplTests.fs (not shared module) — extracting a shared helpers module is separate refactor, scope creep
- Defensive empty-thought render in `Rendering.fs` deferred — schema `minLength: 1` on thought makes empty-thought unreachable; defensive guard would be dead code
- `Rendering.fs` zero code changes — existing `renderVerbose` path already reads `step.Thought`, only inputs become meaningful

### Issues Resolved During Milestone

- **v1.0 UAT hotfix `5ab5a95`** (Router.modelToName absolute path hardcode) replaced with proper dynamic resolution in 06-01
- **`bootstrapAsync` timeout WARN** when `--model 72b` targets 8001 but 8000 is cold — fixed by lazy-per-port probe in 06-02
- **mlx_lm.server HF Hub fallback regression** (discovered mid-milestone via live re-verification) — fixed by 06-03 local-path preference
- **`Step.Thought` placeholder** `"[not captured in v1]"` in `--verbose` output — replaced with actual LLM reasoning via `LlmResponse` plumbing in Phase 7
- **Expecto `Console.SetOut` global-state races** (documented but not re-hit this milestone) — `testSequenced` pattern from v1.0 preserved

### Issues Deferred (v1.2 candidates)

- **Per-port `MaxModelLen` in `AppComponents`** — currently hardcoded 8192 floor; 72B's actual max_model_len is no longer queried at startup (research accepted as known regression for v1.1)
- **Mac unified memory management** — 32B + 72B simultaneous serving on 128GB Mac works fine for normal workloads (confirmed by stress test: 3 blueCode runs @ 2s each, no OOM, prompt cache grew 0.70→0.83GB). Long-term: periodic service restart to clear prompt cache accumulation
- **Multi-platform `tryParseModelId` heuristic** — current `StartsWith("/")` is macOS/Linux only; Windows support (if ever added) needs different path detection
- **Real tokenizer for context window estimation** — still using `chars/4` approximation (carried from v1.0)
- **Streaming, `edit_file`, session persistence** — v1.2+ scope per v1.0 research

### Technical Debt Incurred

- **`AppComponents.MaxModelLen = 8192` constant** — 72B has larger actual max_model_len (e.g., 32K) but probe runs per-port, not surfaced at bootstrap. Accepted regression; per-port visibility is v1.2 work.
- **`makeMockResponse` helper duplicated** in two test files — consolidating to a shared test helpers module deferred to v1.2 test infrastructure pass (if ever; current duplication is minimal)
- **06-03 gap closure revealed mlx_lm.server black-box behavior** — depending on the server always reporting a local-path id in `/v1/models` data array. If mlx_lm.server changes behavior (e.g., stops reporting path id), our heuristic breaks. Not worth defending in v1.1 since we own the deployment.
